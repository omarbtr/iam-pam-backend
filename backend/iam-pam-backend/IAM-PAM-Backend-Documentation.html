1. Vue d'ensemble du projet

1.1 Description générale

IAM-PAM Backend est une plateforme SaaS multi-tenant développée dans le cadre d'un Projet de Fin d'Études à l'ESPRIT. Elle combine deux domaines de cybersécurité complémentaires :

🔐 IAM — Identity and Access Management

Gestion centralisée des identités, des rôles et des droits d'accès pour chaque organisation (tenant). Intégration avec Active Directory/LDAP via Keycloak pour l'import des utilisateurs existants.

🛡️ PAM — Privileged Access Management

Contrôle des accès privilégiés aux ressources sensibles (serveurs SSH, sessions RDP, bases de données, APIs, applications web). Chaque accès passe par un workflow de demande → approbation → expiration avec traçabilité complète.

1.3 Architecture multi-tenant

Chaque organisation cliente (tenant) est parfaitement isolée des autres grâce à une architecture multi-tenant basée sur deux mécanismes :

1
Isolation PostgreSQL par schéma : Chaque tenant possède son propre schéma PostgreSQL (ex : tenant_bank_a). Le schéma shared contient les tables communes (tenants, services, audit_logs).
2
Isolation Keycloak par groupe : Chaque tenant est représenté par un groupe Keycloak. Chaque utilisateur appartient à un groupe-tenant. Le JWT contient le claim groups utilisé pour identifier le tenant actif.
3
TenantContext (ThreadLocal) : Le TenantInterceptor extrait le tenantId du JWT à chaque requête et l'injecte dans un ThreadLocal. Tous les repositories l'utilisent pour filtrer automatiquement les données.
Realm Keycloak : iam-pam-realm  |  Client : iam-pam-backend  |  Port Keycloak : 8080  |  Port Backend : 8081

5. Sécurité & Authentification

5.1 Modèle de rôles Keycloak

Le système utilise 5 rôles de realm Keycloak avec des périmètres clairement définis :

Rôle KeycloakNiveauPérimètre d'accès
admin Plateforme Créer/modifier/supprimer des tenants, gérer les abonnements de services, accès à toute la plateforme
tenant-admin Tenant Configurer le domaine, importer/gérer les utilisateurs, gérer les ressources PAM, approuver les demandes d'accès, consulter les audits
user Utilisateur Consulter les ressources PAM, créer des demandes d'accès, voir ses propres demandes
pam-access Utilisateur PAM Accéder aux ressources PAM approuvées, créer des demandes d'accès
auditor Conformité Lecture seule des logs d'audit (pas de modification possible)

5.2 JWT & Extraction des rôles

L'application est configurée comme OAuth2 Resource Server. Chaque requête doit inclure un header Authorization: Bearer <jwt>.

Issuer URI : http://localhost:8080/realms/iam-pam-realm
JWK Set URI : http://localhost:8080/realms/iam-pam-realm/protocol/openid-connect/certs
Durée du token : 30 minutes (configurable dans Keycloak Realm Settings → Tokens)

La classe SecurityConfig.extractAuthorities() extrait les rôles depuis le claim realm_access.roles du JWT et les préfixe avec ROLE_ pour Spring Security. Exemple : le rôle Keycloak tenant-admin devient ROLE_tenant-admin.

Le claim groups du JWT contient le groupe-tenant de l'utilisateur (ex: ["tenant-bank-a"]). C'est ce claim qui est utilisé par le TenantInterceptor pour établir le contexte tenant.

5.3 TenantInterceptor — Middleware de sécurité

Exécuté avant chaque requête HTTP. Il effectue deux opérations critiques :

1
Établissement du contexte tenant : Extrait le premier groupe du claim JWT groups et l'injecte dans TenantContext (ThreadLocal). Tous les repositories utilisent automatiquement ce contexte pour filtrer les données.
2
Enforcement de la configuration du domaine : Si l'utilisateur a le rôle tenant-admin et que son tenant n'a pas encore configuré son domaine (domainConfigured = false), toute requête hors whitelist retourne : 403 Forbidden avec "setupUrl": "/api/admin/tenants/my/domain"
3
Nettoyage du ThreadLocal : Dans afterCompletion(), le contexte est systématiquement effacé pour éviter les fuites mémoire et les contaminations entre requêtes.

Endpoints toujours accessibles (whitelist) même sans domaine configuré :

  • POST /api/admin/tenants/my/domain — configurer le domaine
  • GET /api/admin/tenants/my/domain/status — vérifier le statut
  • GET /api/admin/tenants/my — obtenir les infos du tenant

5.4 Isolation multi-tenant

L'isolation des données entre tenants est garantie à plusieurs niveaux :

NiveauMécanismeExemple
HTTP TenantInterceptor → TenantContext (ThreadLocal) tenantId = "tenant-bank-a" injecté dans chaque requête
Service Vérification explicite d'appartenance if (!resource.getTenantId().equals(tenantId)) throw SecurityException
Repository Filtrage par tenantId dans toutes les queries findByTenantIdAndIsActiveTrue(tenantId)
DB Schéma PostgreSQL dédié par tenant Schéma tenant_bank_a séparé physiquement
Keycloak Groupe Keycloak par tenant Groupe tenant-bank-a dans Keycloak

6. Workflows fonctionnels

6.1 Workflow — Création & Onboarding d'un tenant

1
[admin] Créer le tenant
POST /api/admin/tenants avec tenantId, tenantName, maxUsers, services[]
→ Tenant créé en DB + groupe Keycloak créé + services assignés
2
[admin] Créer l'utilisateur tenant-admin dans Keycloak
Dans Keycloak Admin Console → Users → Create User → assigner le rôle tenant-admin → ajouter au groupe tenant-bank-a
3
[tenant-admin] Première connexion → Configuration du domaine
Sans domaine configuré, toute requête retourne 403.
POST /api/admin/tenants/my/domain avec {"domain": "bank.iam-pam.com"}
→ domainConfigured = true, accès complet débloqué
4
[tenant-admin] Import des utilisateurs depuis l'AD
Recherche dans l'annuaire → POST /api/admin/users/import
→ User ajouté au groupe Keycloak, rôles assignés, compteur incrémenté

6.2 Workflow — Import d'un utilisateur depuis l'Active Directory

1
Recherche dans l'annuaire
GET /api/admin/users/directory/search?query=john
→ Keycloak Admin API interroge le LDAP fédéré, retourne les utilisateurs trouvés
2
Vérification de la limite
GET /api/admin/users/can-add → vérifie currentUserCount < maxUsers
3
Import de l'utilisateur
POST /api/admin/users/import avec {"username": "john.doe", "roles": ["user", "pam-access"]}
→ Lock pessimiste sur le tenant → Ajout au groupe Keycloak → Assignation des rôles → Incrémentation compteur
4
L'utilisateur peut maintenant se connecter
Il utilise ses credentials AD (SSO via LDAP fédéré) pour obtenir un token Keycloak

6.3 Workflow — Cycle de vie d'une demande d'accès PAM

Prérequis : Le tenant doit avoir souscrit au service PAM. L'administrateur doit avoir créé au moins une ressource PAM.
1
[user / pam-access] Créer une demande
POST /api/pam/requests avec resourceId, justification, durationHours
→ Demande créée en statut PENDING + audit ACCESS_REQUESTED
2
[tenant-admin] Consulter les demandes en attente
GET /api/pam/requests/pending → Liste des demandes PENDING du tenant
3a
[tenant-admin] Approuver la demande
PUT /api/pam/requests/{id}/review avec {"status": "APPROVED", "comment": "OK"}
→ Statut = APPROVED, expiresAt = now + durationHours, audit ACCESS_APPROVED
3b
[tenant-admin] Rejeter la demande
→ Statut = REJECTED, commentaire enregistré, audit ACCESS_REJECTED
4a
[Automatique] Expiration de l'accès
Le scheduler vérifie toutes les 60 secondes les demandes APPROVED dont expiresAt < now
→ Statut = EXPIRED (saveAll batch)
4b
[tenant-admin] Révocation manuelle
PUT /api/pam/requests/{id}/revoke
→ Statut = REVOKED, audit ACCESS_REVOKED
